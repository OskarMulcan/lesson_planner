Agent Context
=============

This file documents the current repository, technology choices, conventions and recent
schema changes so other agents or automated tools can understand the project and act
consistently.

Technologies
------------
- Python 3.14+
- SQLAlchemy ORM (>=2.0.50)
- PostgreSQL (production target) with btree_gist extension
- Pydantic / pydantic-settings for validation and configuration
- Typer for the CLI
- uv as package manager and task runner

Repository layout
-----------------
- src/lesson_planner: main package
  - config.py, logging_setup.py, database.py
  - models.py: SQLAlchemy models
  - imports/: CSV import handlers and run_import implementation
  - cli/: Typer CLI entry points
- data/examples: sample CSVs for imports

Recent changes
--------------
- Schedule.is_active changed from SmallInteger to Boolean. It is now a native PostgreSQL
  boolean column with server_default="false". Use it directly as a boolean in code.
- ScheduleEntryDraft model/table removed. Code and docs have been updated to remove
  references and the table will no longer be created by schema initialization.
- ImportStaging is configured to be an UNLOGGED table in PostgreSQL to reduce WAL
  pressure. The model sets __table_args__ = {"prefetch_rows": 0} and a DDL event
  listener is registered to run `ALTER TABLE import_staging SET UNLOGGED` after
  table creation — the DDL listener is the portable approach used because older
  SQLAlchemy versions do not accept a direct `postgresql_unlogged` table kwarg.
- The import pipeline (imports/base.py) now deletes staging rows for the importer
  session_id after a successful import run. This preserves the staging table structure
  while keeping it empty between runs.

Conventions and agent rules
---------------------------
- Keep code comments to a minimum; prefer descriptive docstrings for public functions
  and classes.
- Avoid special characters and long/em-dashes in identifiers and autogenerated text.
- Use native PostgreSQL features via DDL event listeners when SQLAlchemy lacks a
  high-level construct (document the approach in AGENT_CONTEXT.md).
- Linting and typing: ruff and mypy are used in development; follow their defaults
  unless there is a clear reason to deviate.

How to run quick checks
-----------------------
- Sanity-import of models (no DB required):
  uv run python -c "import sys; sys.path.insert(0, r'./src'); import lesson_planner.models; print('models import OK')"

Running Python code
-------------------
- Always use `uv run` to execute Python scripts to ensure the project's virtual environment
  is activated and dependencies are available:
  uv run python path/to/script.py
- For the CLI tool, use: uv run python -m lesson_planner.cli <command>
- To sync dependencies before running: uv sync

Version control
---------------
- This project uses Git for version control, but automated agents should NOT run git
  commands. The repository maintainer will handle all commits, branching, and merges.
- Agents should focus on code changes, testing, and documentation only.

Notes for future automation
---------------------------
- Consider adding alembic migrations that capture DDL steps (enums, exclusion constraints,
  and the unlogged creation) for reproducible deployments.
